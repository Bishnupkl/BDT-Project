# Kafka And Docker Setup Guide

This guide explains the Bishnu project scope for the real-time cryptocurrency analytics pipeline: Docker setup, Kafka setup, Java Kafka producer, and integration testing.

## What This Part Builds

The current project creates a real-time data entry point for the larger analytics pipeline.

```text
Java Crypto Producer
        |
        v
Apache Kafka topic: crypto-topic
        |
        v
Spark infrastructure ready for a Spark Streaming app
        |
        v
Future storage/dashboard pipeline
```

The producer simulates live cryptocurrency price events and sends JSON messages into Kafka. Spark master and worker containers are now part of Docker Compose, so Student 2 can attach a Spark Structured Streaming application to the same Kafka topic. The actual Spark consumer application is still separate from this Student 1 infrastructure work.

## Crypto Data Source

This project does not call any external cryptocurrency API right now.

There is no CoinMarketCap API, CoinGecko API, Binance API, Coinbase API, or any other HTTP API used in the current producer.

The crypto data is simulated inside the Java producer. This was done intentionally for the Big Data pipeline setup because it makes the project easier to run, test, and demonstrate without needing:

- API keys
- internet access
- rate-limit handling
- external service availability
- paid crypto data plans

Current data flow:

```text
CryptoPriceGenerator.java
        |
        v
CryptoPriceEvent.java
        |
        v
CryptoJsonMapper.java
        |
        v
CryptoPriceProducer.java
        |
        v
Kafka topic
```

### How Crypto Details Are Created

Crypto details are generated by:

```text
producer/src/main/java/edu/miu/bdt/producer/CryptoPriceGenerator.java
```

The generator has a static list of supported crypto profiles:

```text
BTC -> base price 65231.45
ETH -> base price 3210.20
SOL -> base price 145.60
ADA -> base price 0.46
XRP -> base price 0.53
```

Each profile includes:

```text
symbol
basePrice
volatility
```

For every new event, the generator:

1. Randomly selects one crypto profile.
2. Generates a random price movement using the profile volatility.
3. Calculates a simulated price.
4. Rounds the price to 2 decimal places.
5. Adds the current UTC timestamp.
6. Returns a `CryptoPriceEvent`.

The generated object looks like:

```java
new CryptoPriceEvent("BTC", BigDecimal.valueOf(65231.45), Instant.now())
```

Then Jackson converts it to JSON:

```json
{"symbol":"BTC","price":65231.45,"timestamp":"2026-05-08T20:42:33.304012810Z"}
```

```text
        |
        v

        |
        v

        v

        |
        v
Kafka topic
```

Example possible APIs:

```text

CoinMarketCap API
Binance API
Coinbase API
```

Example future HTTP call:

```bash
GET https://api.coingecko.com/api/v3/simple/price?ids=bitcoin,ethereum&vs_currencies=usd
```

Example future response:

```json
{
  "bitcoin": {
    "usd": 65231.45
  },
  "ethereum": {
    "usd": 3210.20

```

That response would then be mapped into our existing Kafka message format:

```json

```

No such API call exists in the current code. The current project is fully local and simulated.

## Docker Compose Services

The main Docker setup is in:

```bash
docker/docker-compose.yml
```

It defines these services:

```text
zookeeper
kafka
kafka-setup
kafka-ui
spark-master
spark-worker
crypto-producer
```

### 1. zookeeper

Kafka uses Zookeeper in this setup for broker coordination.

```yaml
zookeeper:
  image: confluentinc/cp-zookeeper:7.6.1
```

Important settings:

```text
ZOOKEEPER_CLIENT_PORT=2181
ZOOKEEPER_TICK_TIME=2000
```

It exposes port:

```text
2181
```

### 2. kafka

This is the Kafka broker. It stores messages and manages topics.

```yaml
kafka:
  image: confluentinc/cp-kafka:7.6.1
```

Important ports:

```text
9092  -> used from the host machine
29092 -> used by containers inside Docker network
```

Important advertised listeners:

```text
PLAINTEXT://kafka:29092
PLAINTEXT_HOST://localhost:9092
```

Why both are needed:

- `kafka:29092` lets Docker containers talk to Kafka.
- `localhost:9092` lets your local machine talk to Kafka.

Kafka healthcheck:

```bash
kafka-broker-api-versions --bootstrap-server localhost:9092
```

This makes Docker wait until Kafka is ready before running dependent services.

### 3. kafka-setup

This service creates the main Kafka topic automatically:

```text
crypto-topic
```

It runs:

```bash
kafka-topics --bootstrap-server kafka:29092 \
  --create \
  --if-not-exists \
  --topic crypto-topic \
  --partitions 1 \
  --replication-factor 1
```

This means you do not need to manually create the topic every time.

### 4. kafka-ui

Kafka UI provides a browser interface for Kafka topics and messages.

```yaml
kafka-ui:
  image: provectuslabs/kafka-ui:latest
```

Access it at:

```text
http://localhost:8085
```

It connects to Kafka inside Docker using:

```text
kafka:29092
```

Use it to:

- view topics
- inspect messages
- verify producer activity
- help validate Spark/Kafka integration

### 5. spark-master

Spark master coordinates Spark workers and Spark jobs.

```yaml
spark-master:
  image: spark:3.5.8
```

Important ports:

```text
8080 -> Spark UI
7077 -> Spark master endpoint
```

Access Spark UI at:

```text
http://localhost:8080
```

Spark jobs can use:

```text
spark://spark-master:7077
```

### 6. spark-worker

Spark worker connects to Spark master and executes Spark tasks.

```yaml
spark-worker:
  image: spark:3.5.8
```

Important command:

```text
spark-class org.apache.spark.deploy.worker.Worker spark://spark-master:7077
```

### 7. crypto-producer

This service builds and runs the Java producer from:

```text
producer/
```

It depends on `kafka-setup`, so it only starts after Kafka is healthy and the topic exists.

Important environment variables:

```text
KAFKA_BOOTSTRAP_SERVERS=kafka:29092
KAFKA_TOPIC=crypto-topic
PRODUCER_INTERVAL_MS=1000
PRODUCER_MAX_EVENTS=0
```

Meaning:

- `KAFKA_BOOTSTRAP_SERVERS`: Kafka broker address.
- `KAFKA_TOPIC`: topic where messages are sent.
- `PRODUCER_INTERVAL_MS`: delay between messages.
- `PRODUCER_MAX_EVENTS`: number of messages to send. `0` means keep running forever.

The producer service is behind a Docker Compose profile:

```yaml
profiles:
  - producer
```

So it does not run automatically unless you request it.

## Shared Network, Volume, And Env

All Docker services use the same bridge network:

```yaml
networks:
  crypto-network:
    driver: bridge
```

This lets services resolve each other by name. For example:

```text
spark-master -> kafka
spark-worker -> spark-master
crypto-producer -> kafka
kafka-ui -> kafka
```

Inside Docker, Kafka should be reached at:

```text
kafka:29092
```

From the host machine, Kafka should be reached at:

```text
localhost:9092
```

Kafka data is persisted in a named volume:

```text
kafka-data -> /var/lib/kafka/data
```

This keeps topics and messages available across normal container restarts. To delete the persisted data, run:

```bash
docker compose -f docker/docker-compose.yml down -v
```

The root `.env` file centralizes common settings:

```env
KAFKA_TOPIC=crypto-topic
KAFKA_BOOTSTRAP_SERVERS=kafka:29092
PRODUCER_INTERVAL_MS=1000
PRODUCER_MAX_EVENTS=0
```

The helper scripts load this file with `--env-file`.

## Java Producer

The Java producer code lives here:

```text
producer/src/main/java/edu/miu/bdt/producer/
```

### CryptoPriceEvent.java

Defines the JSON event structure:

```json
{
  "symbol": "BTC",
  "price": 65231.45,
  "timestamp": "2026-05-07T12:00:00Z"
}
```

Fields:

```text
symbol    -> crypto symbol, such as BTC, ETH, SOL, ADA, XRP
price     -> simulated current price
timestamp -> UTC timestamp when event was generated
```

### CryptoPriceGenerator.java

Generates fake live crypto prices.

Supported symbols:

```text
BTC
ETH
SOL
ADA
XRP
```

Each symbol has a base price and volatility. The generator randomly picks a symbol and creates a price around its base value.

### CryptoJsonMapper.java

Configures Jackson JSON serialization.

Important behavior:

- Serializes Java `Instant` as ISO timestamp.
- Avoids numeric timestamp arrays.

Example:

```json
"timestamp":"2026-05-08T20:42:33.304012810Z"
```

### CryptoPriceProducer.java

Main producer application.

It:

1. Reads environment variables.
2. Connects to Kafka.
3. Generates crypto price events.
4. Converts each event to JSON.
5. Sends the JSON message to Kafka.
6. Flushes the producer after each send.
7. Stops after `PRODUCER_MAX_EVENTS`, unless that value is `0`.

Kafka producer settings include:

```text
acks=all
client.id=crypto-price-producer
key.serializer=StringSerializer
value.serializer=StringSerializer
```

The Kafka message key is the crypto symbol.

Example:

```text
key: BTC
value: {"symbol":"BTC","price":65231.45,"timestamp":"..."}
```

## Troubleshooting

### SLF4J warning

If you see:

```text
SLF4J: Failed to load class "org.slf4j.impl.StaticLoggerBinder"
```

Rebuild the producer image:

```bash
docker compose -f docker/docker-compose.yml build crypto-producer
```

The project includes `slf4j-simple`, so a fresh build should remove this warning.

### Old messages are showing

If you consume from `crypto-topic --from-beginning`, Kafka shows old messages first. That is normal.

For clean verification, use:

```bash
bash scripts/integration-test-bishnu.sh
```

It creates a fresh temporary topic.

### Kafka container is not healthy

Check logs:

```bash
docker logs crypto-kafka
```

Restart cleanly:

```bash
docker compose -f docker/docker-compose.yml down
docker compose -f docker/docker-compose.yml up -d zookeeper kafka kafka-setup
```

### Port already in use

Kafka uses:

```text
9092
29092
2181
```

If those ports are busy, stop the other process or update `docker/docker-compose.yml`.
